03: Using Classes in C++

COMP2811 User Interfaces

Nick Efford

Objectives

  • To see how classes are defined and used in C++
  • To compare and contrast this with Java’s approach

Class Definition Comparison

Java

  • Entire class definition in a single file

  • Compiler will generate one .class file for every class defined in the .java file

  • Accessibility specified explicitly for individual fields and methods

C++

  • Class definition often split across two files

    • Header file (always needed)
    • Implementation file

  • Compiler generates one file of object code, regardless of how many classes appear in the implementation file

  • Class definition divided into different sections with different levels of accessibility

  • Methods can be inlined, for performance

  • Methods that don’t modify object state must be declared as const

Example

  • We want to represent circles with a given radius and origin

  • We want to be able to query these attributes

  • We would also like to be able to compute area & perimeter

  • … and test whether a given point is inside a circle

circle.hpp

#pragma once

#include <cmath>

class Circle
{
  public:
    Circle(double r): Circle(0, 0, r) {}
    Circle(double, double, double);
    double getX() const { return xorigin; }
    double getY() const { return yorigin; }
    double getRadius() const { return radius; }
    bool containsPoint(double, double) const;
    double area() const { return M_PI * radius * radius; }
    double perimeter() const { return 2.0 * M_PI * radius; }

  private:
    double xorigin, yorigin;
    double radius;
};
1
Prevents problems with repeated inclusion
2
public section lists the things accessible to class users (constructors & methods)
3
private section lists the things not directly accessible to class users (typically fields)
4
Don’t forget the semicolon after the closing brace!

Use of #pragma

  • We need to prevent class/struct definitions or function prototypes being made multiple times

  • Traditional approach involves include guards:

    #ifndef _CIRCLE_HPP_
#define _CIRCLE_HPP_
class Circle
{
  ...
};
#endif

  • #pragma approach requires only one line instead of three!

circle.hpp

#pragma once

#include <cmath>

class Circle
{
  public:
    Circle(double r): Circle(0, 0, r) {}
    Circle(double, double, double);
    double getX() const { return xorigin; }
    double getY() const { return yorigin; }
    double getRadius() const { return radius; }
    bool containsPoint(double, double) const;
    double area() const { return M_PI * radius * radius; }
    double perimeter() const { return 2.0 * M_PI * radius; }

  private:
    double xorigin, yorigin;
    double radius;
};
5
Like Java, constructors have same name as class and no return type
6
None of the methods modify the state of a Circle object, so they are declared const
7
Simple methods like getters are inlined into the class definition
8
More complex methods appear only as prototypes, and are implemented in the .cpp file

Inlining

  • Simple methods can be implemented directly in the header file, as part of the class definition

  • This is an optimization hint to the compiler – a suggestion that it can substitute the code of the method body in all places where the method is called

  • Potentially a useful way to avoid the normal overhead associated with calling a method or function

  • Compiler is free to ignore the suggestion!

Constructors

Circle(double r): Circle(0, 0, r) {}   // C++11

Circle(double, double, double);

  • Single-parameter constructor:

    • Requires only the radius to be supplied

    • Is inlined, and delegates the work to three-parameter constructor, setting x & y to 0

  • Three-parameter constructor:

    • Requires values for x, y and radius

    • Is not inlined; implementation appears in .cpp file

circle.cpp

#include <stdexcept>
#include "circle.hpp"

Circle::Circle(double x, double y, double r):
  xorigin(x), yorigin(y), radius(r)
{
  if (radius <= 0.0) {
    throw std::invalid_argument("radius must be > 0");
  }
}

bool Circle::containsPoint(double x, double y) const
{
  double dx = x - xorigin;
  double dy = y - yorigin;
  return sqrt(dx*dx + dy*dy) <= radius;
}
1
Need this header if we want to use standard C++ exceptions
2
Need to include the class definition in .cpp file
3
Circle:: prefix indicates that this is a member of the Circle class
4
Initializer list is used to assign parameter values to fields
5
C++ exceptions are thrown in a similar way to Java, but note no use of new to create the object!
6
const qualifier needs to be used here as well as in the class definition

Initializer Lists

Could do this:

Circle::Circle(double x, double y, double r)
{
  xorigin = x;
  yorigin = y;
  radius = r;
}

But compiler generates more efficient code for this:

Circle::Circle(double x, double y, double r):
  xorigin(x), yorigin(y), radius(r)
{
}

Using The Class

// Stack allocation

Circle a(2.5);
Circle b { 2.5 };  // C++11 onwards

Circle c(-5.0, 10.0, 8.5);
cout << "Area = " << c.area() << endl;

// Heap allocation

Circle* d = new Circle(1.2, -3.5, 12.7);
cout << "Perimeter = " << d->perimeter() << endl;

Compiling

Often handled on Unix-like systems using Make tool:

Makefile
circdemo: circdemo.o circle.o
        g++ -o circdemo circdemo.o circle.o

circdemo.o: circdemo.cpp circle.hpp
        g++ -c circdemo.cpp

circle.o: circle.cpp circle.hpp
        g++ -c circle.cpp

  • First line of a rule specifies target, then the things it depends on

  • Subsequent lines of rule, prefixed with Tabs, are commands needed to build the target from its dependencies

Alternatively…

Higher-level approach uses CMake:

CMakeLists.txt
cmake_minimum_required(VERSION 3.6)
project(circles LANGUAGES CXX)

set(SOURCE_FILES
        circle.hpp
        circle.cpp
        circdemo.cpp)

add_executable(circdemo ${SOURCE_FILES})

Note

CMake is a meta-build tool. Running it on Unix-like systems will typically generate a makefile, which can then be used to build the registered executables…

Inheritance

shape.hpp

#pragma once

class Shape
{
  public:
    Shape(double x, double y): xorigin(x), yorigin(y) {}
    double getX() const { return xorigin; }
    double getY() const { return yorigin; }

  private:
    double xorigin;
    double yorigin;
};

Note

All code inlined into class definition, so no .cpp file!

circle2.hpp

#pragma once

#include <cmath>
#include "shape.hpp"

class Circle: public Shape
{
  public:
    Circle(double r): Circle(0, 0, r) {}
    Circle(double, double, double);
    double getRadius() const { return radius; }
    bool containsPoint(double, double) const;
    double area() const { return M_PI * radius * radius; }
    double perimeter() const { return 2.0 * M_PI * radius; }

  private:
    double radius;
};
1
Need to include definition of Shape class
2
C++ uses a colon and access specifier instead of Java’s extends keyword

circle2.cpp

#include <stdexcept>
#include "circle2.hpp"

Circle::Circle(double x, double y, double r):
  Shape(x, y), radius(r)
{
  if (radius <= 0.0) {
    throw std::invalid_argument("radius must be > 0");

  double dy = y - getY();

Initializer list delegates to superclass constructor
2
xorigin private so getX() is required here

Dynamic Binding

  • In Java, dynamic binding of overridden method calls is the default, whereas in C++ static binding is the default

  • Static binding is enabled in Java by making a method final

  • Dynamic binding in enabled in C++ by

    • Declaring the overridden method to be virtual
    • Referencing the object using a pointer

Example

// Abstract superclass

class Shape
{
  public:
    ...
    virtual void draw() const = 0;
  ...
};
1
= 0 indicates this is a ‘pure virtual method’ (one with no valid implementation in superclass) 
// Concrete subclass

class Circle: public Shape
{
  public:
    ...
    virtual void draw() const;
  ...
};
2
No need to repeat virtual here (but clearer if we do)

Example

vector<Shape*> shapes;
shapes.push_back(new Circle(2.5));
shapes.push_back(new Rectangle(0.0, 1.0, 11.5, 5.2));
shapes.push_back(new Circle(-1.6, 7.2, 3.5));
...
for (Shape* s: shapes) {
  s->draw();
}
1
We could represent a picture as a collection of Shape pointers, allowing for a mixture of types of shape
2
C++11 standard supports a ‘for each’ style loop, like Java
3
Dynamic binding occurs here, invoking the appropriate draw() on whichever type of object the pointer is referencing

Polymorphism

Loop on lines 6–8 is an example of polymorphic code; it will work with any type of shape, including classes that haven’t yet been created
